| File | Responsibility | Notes |
| --- | --- | --- |
| `server.js` | Main HTTP server, app-server client, auth, routes, thread detail compaction, target-turn Usage summary backfill from rollout scans, uploads, continuation jobs, Web Push, update/restart endpoints, thread visibility filtering including migrated Windows-cwd history on macOS, rollout-session fallback time/status inference, local-file preview roots, authenticated GitHub link-preview route, app-server thread goal set/action/list/detail decoration, inherited runtime model/effort application, server-side cross-thread task-card draft materialization from completed turns/thread-detail reads, server-side thread-display mode/pane-slot/split-pair runtime settings, the thread-callable direct task-card route gated by the runtime `跨工作区委派` setting, Codex app-server dynamic tool injection/`item/tool/call` handling for `codex_mobile.delegate_to_thread` and `codex_mobile.return_to_source`, automatic per-Codex-Home `codex_mobile` MCP toolset registration, workspace-delegation sandbox/profile wiring plus app-server approval decisions for dynamic source-write guard, and the disabled `workspace-delegation` compatibility route | Treat as composition glue. Extract reusable logic instead of expanding large inline blocks. |
| `codex-app-server-mux.js` | Shared Desktop/Mobile app-server bridge, endpoint publication, TCP server, app-server notification replay, approval proxying | Stdout is protocol data for Desktop; diagnostics must go to mux log. When it starts the real Codex CLI child, it must drop Desktop-bridge-only `CODEX_CLI_PATH` and `CODEX_MUX_*` env vars. |
| `codex-app-server-mux-shim.cs` | Windows `.exe` shim for Desktop `CODEX_CLI_PATH` | Rebuild/relaunch Desktop through the shared launcher after changes. The default generated shim is `codex-app-server-mux-win.exe` so older running `codex-app-server-mux.exe` processes do not block rebuilds. The shim itself must be compiled as `/target:winexe`; shim-spawned Node must use `CREATE_NO_WINDOW` and hidden startup flags so Desktop bridge helpers never open a console. |
| `start-codex-mobile-web*.ps1/.vbs` | Windows startup wrappers and hidden scheduled-task startup | User-logon task is preferred when WSL access is needed; LocalSystem `AtStartup` mode is supported for no-login boot and must map `USERPROFILE`/`HOME`/`APPDATA`/`LOCALAPPDATA`/`TEMP`/`TMP` back to the target user before resolving Codex state or binaries. Plugin HTTPS base URL, Hermes frame-origin settings, active Codex profile resolution, profile auth/config backup without replacement, non-default profile shared-state hardlink/junction setup, and clearing Desktop bridge env before Mobile-owned CLI startup must be preserved here for scheduled deployments. |
| `restart-codex-mobile-shared-chain.ps1` | Scoped restart for Mobile Web chain | Resolves the active profile `CODEX_HOME` before removing/waiting on the mux endpoint. Does not restart Hermes Mobile, WSL, Codex Desktop, or unrelated services. |
| `scripts/deploy-macos-plugin.ps1` | Windows-orchestrated Mac Hermes plugin production deploy harness | Deploys only a clean public git archive, rejects private/runtime paths, stages checks before target backup/sync, runs target checks before LaunchDaemon restart, and smokes status without printing key material. Keep covered by `test/macos-plugin-deploy-harness.test.js`. |
| `scripts/create-thread-task-card.js` | Local UTF-8-safe CLI wrapper for `POST /api/threads/:sourceThreadId/task-cards`, intended for Codex-thread/tool initiated delegation | Reads the Codex Mobile access key from env or runtime key file, never prints it. Direct source-thread approval is gated by the runtime `跨工作区委派` setting; otherwise cards stay pending. Supports bounded `--reasoning-effort` for deep audit cards. |
| `scripts/return-thread-task-card.js` | Local UTF-8-safe CLI wrapper for `POST /api/thread-task-cards/:id/reply`, intended for target-thread manual return cards | Reads the Codex Mobile access key from env or runtime key file, never prints it, creates a stable `task-card-return:*` idempotency key when the caller does not supply one, and marks replies as `returnToSource` so they are source-direct approved instead of ordinary pending replies. |
| `scripts/codex-mobile-mcp-server.js` | Codex-thread stdio MCP server exposing `list_threads`, `delegate_to_thread`, and `return_to_source` | Calls the authenticated local Codex Mobile HTTP API and never stores or prints raw key material. Registered automatically into all known/target Codex Homes by `syncKnownCodexMobileMcpToolsets()` and `syncCodexMobileMcpToolset()`, including tool-level `approval_mode = "approve"` entries. Exposes bounded `reasoningEffort` on delegation and `partially_completed` return status. |
| `start-codex-desktop-shared.ps1` / `start-codex-desktop-shared-hidden.vbs` / `start-codex-desktop-*.cmd` | Windows Desktop shared-mux launcher and profile escape hatch | Use `-ProfileId default|current|previous` or `-CodexHome` so Desktop attaches to the same selected profile endpoint as Mobile Web; preserve non-default profile auth/config backup and shared thread/workspace state hardlink/junction setup. When no real Codex exe is explicit, prefer the newest installed OpenAI Codex bin before the older runtime copy. Desktop shortcuts should target the VBS wrapper or another hidden host, not the `.cmd` wrappers. Only the Desktop GUI may be visible; bridge helper processes must be windowless. |
| `start-codex-*-macos.sh` | macOS standalone/shared launch helpers | Keep shell syntax validated with `npm run check:macos`. |

## Ownership Boundaries

- `server.js` owns HTTP protocol shape and app-server request sequencing, but domain-specific state decisions should live in adapters.
- `codex-app-server-mux.js` owns shared-stream mechanics. Do not move browser UI logic into the mux.
- `public/sw.js` owns only cache and Push behavior. Do not put app state or route logic there.
- `.agent-context/` owns durable local context. Do not copy it to public release.
- Runtime files under `%USERPROFILE%\.codex-mobile-web` and `%USERPROFILE%\.codex` are not source files.

## Adding A New Module

1. Prefer `adapters/<domain>-service.js` for server-side policy or state logic.
2. Prefer `public/<domain>.js` for reusable browser helpers.
3. Add focused `node --test` coverage close to the behavior.
4. Add the new JS file to `package.json` `check` if it should be syntax-checked.
5. Document the module here when it becomes part of a stable boundary.
