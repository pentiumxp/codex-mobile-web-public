### Thread Detail Active Overlay Invariant

The active-overlay path may avoid foreground `thread/read` only after
server-side proof shows the live active turn is complete enough for display.
`services/thread-detail/thread-detail-active-overlay-provider-service.js` classifies memory-only live
snapshots as `full` or `partial`; notification-shell snapshots and snapshots
without rollout signature evidence are partial. `services/thread-detail/thread-detail-active-window-overlay-policy-service.js`
rejects partial live snapshots until read orchestration backfills them from a
bounded active-window projection or falls through to full `thread/read`.

`thread-detail-read-orchestration-service` can pre-probe the dedicated
active-overlay projection-window lookup when the summary missed active state.
That hot path may return `projection-active-overlay` only if the provider
evidence is already full, or if a partial live overlay is merged with an
initial/bounded active-window backfill and then re-proven as `backfilled`.
This prevents a notification-tail-only active turn from replacing the full
visible in-progress turn while preserving the warm active-detail path.

Current render-only exceptions include the first-paint visible budget notice:
`public/app.js` renders `mobileVisibleItemBudget` / `mobileOmittedVisibleItemCount`
as a compact non-`data-item` notice in single-thread and tiled panes, and adds
the bounded budget summary to the conversation signature so server-budgeted
omissions are not diagnosed as client-side lost messages. The budget policy
itself belongs to `services/thread-detail/thread-detail-response-budget-service.js`;
the adapter path remains only as a compatibility export.

## Ownership Boundaries

- `server.js` owns HTTP protocol shape and app-server request sequencing, but domain-specific state decisions should live in adapters.
- `codex-app-server-mux.js` owns shared-stream mechanics. Do not move browser UI logic into the mux.
- `public/sw.js` owns only cache and Push behavior. Do not put app state or route logic there.
- `.agent-context/` owns durable local context. Do not copy it to public release.
- Runtime files under `%USERPROFILE%\.codex-mobile-web` and `%USERPROFILE%\.codex` are not source files.

## Adding A New Module

1. Prefer `adapters/<domain>-service.js` for server-side policy or state logic.
2. Prefer `public/<domain>.js` for reusable browser helpers.
3. Add focused `node --test` coverage close to the behavior.
4. Add the new JS file to `package.json` `check` if it should be syntax-checked.
5. Document the module here when it becomes part of a stable boundary.
