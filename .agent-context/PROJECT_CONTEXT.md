Use `-UserId <domain\user> -UserProfilePath C:\Users\<user>` when installing from another account or SYSTEM/elevated automation context.
- `-RunAsSystem` remains available only as an explicit optional mode for startup before user logon or survival after sign-out, but that mode cannot launch WSL distributions.
- The windowless startup wrapper is a small supervisor: if the 8787 Node listener exits, it waits briefly and restarts it while reusing the existing mux endpoint. It sets `HOME`, `HOMEDRIVE`, `HOMEPATH`, `CODEX_HOME`, and `CODEX_MOBILE_RUNTIME_DIR` for the target Windows user profile. Under the optional SYSTEM task it also injects process-local Git `safe.directory` entries for the user's Documents tree and Codex runtime dirs so user-owned repos do not fail with dubious ownership inside Codex tool runs.

Shared-chain restart is now manual through the mobile sidebar `Restart` button next to the version/update pill, which asks for confirmation and calls authenticated `POST /api/restart/shared-chain`. The endpoint launches `restart-codex-mobile-shared-chain.ps1` after responding; the script stops only the Codex Mobile Web shared chain (`Codex Mobile Web` task, hidden/windowless launchers, this workspace's `server.js`, this workspace's app-server mux, and `%USERPROFILE%\.codex-mobile-web\codex.exe app-server`), removes the stale mux endpoint file, restarts `Codex Mobile Web`, and waits for HTTP plus mux readiness. It intentionally does not restart Hermes Mobile, Gateway, WSL, or Codex Desktop. Manual restart logs to `%USERPROFILE%\.codex-mobile-web\shared-chain-restart.log`.
- Mobile Web has a self-update check for Git checkouts. On startup and after browser login it can fetch `CODEX_MOBILE_UPDATE_REMOTE` / `CODEX_MOBILE_UPDATE_BRANCH` (default `origin/main`), show an update pill in the sidebar version area, and apply only clean fast-forward updates after user confirmation. The apply path refuses dirty worktrees, branch mismatch, ahead/diverged branches, and non-Git installs, then exits the Node listener so the existing startup supervisor restarts from updated files. Runtime state under `%USERPROFILE%\.codex-mobile-web` is outside the repo and is not overwritten.
- Self-update restart requires a supervisor such as the Windows hidden startup wrapper, a user-logon Scheduled Task using that wrapper, or the macOS shared launcher. Manual `node server.js`, `npm start`, or one-shot shell deployments will update tracked files and then stop until the operator manually restarts the service.
- The Windows startup task defaults to `-EnsureStandaloneMux -RequireSharedAppServer`, preserving the single shared app-server stream while avoiding a silent managed app-server fallback. It starts a standalone mux endpoint when no live mux endpoint exists, and Codex Desktop can later attach to that endpoint through `start-codex-desktop-shared.ps1`. Use `-AllowManagedFallback` only for intentional standalone Mobile Web operation.
- Current Tailscale HTTPS mapping is `https://gmk.tail62e8ce.ts.net:8443/ -> http://127.0.0.1:8787`.
- The browser authenticates with the access key, then receives live updates through Server-Sent Events.
- Web Push support stores VAPID keys in `%USERPROFILE%\.codex-mobile-web\web-push-vapid.json` and subscriptions in `%USERPROFILE%\.codex-mobile-web\web-push-subscriptions.json`; neither file should be committed or copied into shared context. The browser must access Codex Mobile Web through HTTPS, currently Tailscale Serve `https://gmk.tail62e8ce.ts.net:8443/`, to subscribe. The VAPID subject must be a non-localhost contact URI; Apple Push rejects localhost subjects with `BadJwtToken`.

It caches `/`, `/index.html`, `/styles.css`, `/app.js`, `/manifest.json`, `/api-client.js`, `/runtime-settings.js`, `/draft-store.js`, `/markdown-renderer.js`, and public icon assets, bypasses `/api/` and `/uploads/`, and uses network-first fetches with cached fallback. Page refresh prompts must only be shown after the browser has fetched and populated the target shell cache with the complete app-shell asset set; clicking the prompt must repeat that check before pruning old `codex-mobile-shell-*` caches or reloading. Private now follows the public `sw.js` path instead of maintaining a separate `public/service-worker.js`.
- Web Push turn-completed notifications use the completed thread title as the notification title and include `This turn 已结束 · <local time>` in the body. They bind `turn/started` metadata by turn id and reuse that thread id/title on `turn/completed`, then include a `/?thread=<threadId>` target URL in notification data. Completion Web Push and the normal thread list must skip Sub Agent child threads recorded in `state_5.sqlite` `thread_spawn_edges` or marked in `threads.agent_nickname` / `threads.agent_role`, while still allowing the parent/main thread completion notification and normal parent/main thread rows. `turn/started` may temporarily observe a turn before thread id is known, but `turn/completed` must fail closed unless it can resolve and classify a known main thread id; missing ids, unknown thread ids, classifier lookup failures, and Sub Agent classifications must not send Web Push. This prevents Sub Agent completions from surfacing as UUID-title Push notifications that cannot deep-link to a real main thread. Notification clicks should focus/open Mobile Web and load that thread directly, even if it is not present in the first rendered thread list. The service worker posts a `codex-open-thread` message to already-open Mobile Web windows. - `codex-app-server-mux.js` can act as the shared bridge: Desktop launches generated `codex-app-server-mux.exe` via `CODEX_CLI_PATH`, the shim starts `node codex-app-server-mux.js`, the mux starts the real app-server over stdio, and Mobile Web connects to the mux endpoint file.

- The mux records `turn/start` requests from Mobile Web and, after the app-server emits `turn/started`, broadcasts a synthetic `mux/userMessage` item to Desktop/Mobile clients so mobile-started new-turn user messages are visible in Codex Desktop.
- The mux proxies app-server server requests, including command, file-change, and permission approvals, so Mobile Web can display approval controls and answer them through the same shared stream.
- The mux keeps a bounded replay buffer for recent app-server `turn/*`, `item/*`, `thread/*`, and rate-limit notifications. Historical notification replay is enabled for both Mobile Web and Desktop by default after `initialize`; set `CODEX_MUX_REPLAY_DESKTOP_NOTIFICATIONS=0` only when diagnosing Desktop reconnect rollback behavior. Unresolved approval/server requests are replayed to all clients.
- Mobile Web declares `CODEX_MOBILE_MUX_REPLAY_NOTIFICATION_LIMIT` (default `200`) when connecting to the mux so reconnects do not replay the entire buffered notification history on long sessions. Unresolved server requests are still replayed separately.
- For Mobile Web clients, the mux drops live command/file output deltas and reasoning text deltas and truncates very large string payloads when sending to the mobile client. Replay buffer entries are stored as full notifications so Desktop reconnect replay keeps full fidelity.
- Mobile Web SSE clients pass their current `threadId` to `/api/events`. The server filters turn/item notifications to that thread, while still allowing status, rate-limit, and thread-list-level notifications through. The server also drops `turn/diff/*` notifications entirely because the mobile UI does not render diffs and large diff payloads can stall the browser before client-side thread filtering runs.
- The mux treats TCP write backpressure as a normal `drain` condition and must not disconnect Desktop/Mobile Web just because one `socket.write()` returns `false`.
- Mobile Web treats a detected mux/shared endpoint as required for that process lifetime. If the shared endpoint disconnects or becomes unavailable, Mobile Web reports the shared app-server error instead of falling back to a managed child and creating a divergent stream.
- Mobile Web must also detect shared mux endpoint drift while its old JSONL TCP socket remains open. Before using the app-server connection, `server.js` compares the live endpoint against `%USERPROFILE%\.codex\app-server-mux\endpoint.json`; if host/port/url changed, it closes the stale socket and reconnects to the current mux instead of staying split from Desktop/current app-server traffic.
- `CODEX_MOBILE_REQUIRE_SHARED_APP_SERVER=1` forces the same no-fallback behavior even when Mobile Web starts before any mux endpoint file exists.
- README documents the app-server bridge as implemented and verified on Windows; macOS Desktop bridge support is documented as an implementation plan only and remains unverified until tested on a real Mac.
- When Mobile Web connects to an already-initialized shared app-server through mux, `initialize` may return `Already initialized`; this is valid for the second client and should be treated as a usable shared connection.
- When Mobile Web sends extra input while a turn is already active, the browser posts the active turn id and `server.js` should use app-server `turn/steer` with `expectedTurnId` so Desktop and Mobile stay on the same active turn stream. After a successful `turn/steer`, `server.js` should also emit a deterministic mux-local `mux/userMessage` echo keyed by `clientSubmissionId`, because current app-server builds can accept the steering request without replaying a visible user-message item.
- Existing-thread message submission must preflight stale active turns before `turn/steer`. If the browser's active turn has been superseded by newer durable turns, is missing from recent turn history after the terminal-idle window, has a quiet rollout for `CODEX_MOBILE_STALE_ACTIVE_TURN_MS` (default 180s), or is quiet for `CODEX_MOBILE_TERMINAL_IDLE_ACTIVE_TURN_MS` (default 45s) after a terminal boundary such as `contextCompaction` or a completed command/tool/file/search item, and there are no pending items or server requests, `server.js` should interrupt that active turn and fall through to `thread/resume` + `turn/start` so new mobile input becomes a durable new turn instead of disappearing into a fake-active app-server turn.
- For new turns and explicit thread resumes, `server.js` should read the thread's latest rollout `turn_context` plus `state_5.sqlite` metadata and forward inherited approval policy, sandbox policy, reasoning summary, and configured verbosity where the app-server protocol supports those fields.
- When local SQLite metadata is unavailable under the optional SYSTEM startup task, runtime-setting lookup should reuse app-server thread summaries and scan large rollout files backward for the latest `turn_context`; large command/output lines can push the relevant context far beyond a small tail read.
- SQLite metadata reads use `adapters/sqlite-cli.js`, which honors `CODEX_MOBILE_SQLITE3_EXE`, then checks common user-local Platform Tools / WinGet `sqlite3.exe` paths before falling back to `sqlite3` on `PATH`. This matters for hidden/scheduled Node listeners whose PATH may not match the interactive PowerShell session; Web Push Sub Agent suppression depends on reading `state_5.sqlite` spawn edges and agent metadata.
- Runtime-setting lookup caches large rollout `turn_context` scan results briefly, keyed by rollout path, size, and mtime, to avoid repeated multi-hundred-MB backward scans during rapid refresh/send paths.
- Thread summaries and detail responses include rollout JSONL size metadata when the rollout file is available. The default warning threshold is `CODEX_MOBILE_ROLLOUT_WARNING_BYTES=209715200` (`200MB`).
- Thread detail reads skip expensive full app-server `thread/read` once rollout size reaches `CODEX_MOBILE_THREAD_DETAIL_ROLLOUT_MAX_BYTES` (default `33554432` / `32MB`) and then prefer bounded `thread/turns/list` before falling back to local summary. This read-performance threshold is intentionally lower than the `200MB` UI warning threshold, so large sessions can still open quickly without showing a warning. Normal-size detail reads prefer full `thread/read` with `CODEX_MOBILE_FULL_THREAD_TURNS` (default `80`, capped at `200`) and fall back to `thread/turns/list` if full read fails. Detail RPC timeouts no longer reset the whole app-server connection.
- `POST /api/thread-continuations` is the preferred browser-facing rollout "压缩续接" action. It creates an in-memory background job and returns `202` with a `jobId`; `GET /api/thread-continuations/<jobId>` reports queued/running/done/error step status so the phone can switch sessions or refresh and resume polling. Jobs are retained according to `CODEX_MOBILE_CONTINUATION_JOB_TTL_MS` and `CODEX_MOBILE_CONTINUATION_JOB_MAX`.
- The older `POST /api/threads` direct continuation path still exists for compatibility, but the browser should use continuation jobs so it does not have to open or block on the source thread first.
- Continuation jobs first compact an oversized workspace `.agent-context/HANDOFF.md` before asking the source thread to write its handoff. `adapters/continuation-handoff-compaction-service.js` archives the full file under `.agent-context/archive/context-compaction-<timestamp>/HANDOFF.full.md` and rewrites the active handoff to a concise current-context file when it exceeds `CODEX_MOBILE_CONTINUATION_CONTEXT_HANDOFF_COMPACT_BYTES` (default `300KB`), preserving recent context according to `CODEX_MOBILE_CONTINUATION_CONTEXT_HANDOFF_PRESERVE_CHARS` (default `60000`). This prevents future continuation bootstraps from repeatedly inheriting multi-MB historical handoff files.
- Continuation jobs then start a source-thread handoff turn that must write `.agent-context/thread-handoffs/<id>.md` in the current workspace, wait for that file, optionally wait longer via `CODEX_MOBILE_CONTINUATION_LATE_HANDOFF_TIMEOUT_MS`, reuse recent handoff files within `CODEX_MOBILE_CONTINUATION_REUSE_HANDOFF_MS`, wait briefly for source turn completion, then start the same-workspace continuation thread through `thread/start`, send a scoped bootstrap message, optionally archive the source thread, and return the new thread id so the browser can switch directly. The handoff directory gets a `.gitignore` to keep generated runtime handoff Markdown/JSONL out of commits.
- The rollout continuation bootstrap must stay file-first and bounded. It carries source thread id/title/cwd/rollout path/rollout size/status, inherited runtime settings, the source-thread-generated handoff file path, bounded handoff excerpts, bounded continuation lineage, recent visible source-turn summaries, `.agent-context/PROJECT_CONTEXT.md` excerpts, and `.agent-context/HANDOFF.md` latest tail. Defaults intentionally keep the bootstrap around an index/excerpt size: `CODEX_MOBILE_CONTINUATION_BOOTSTRAP_CHARS=52000`, source handoff excerpt `12000`, workspace project context `18000`, workspace handoff tail `18000`, item summary `1200`, and non-user items per turn `4`. Continuation success appends `.agent-context/thread-handoffs/index.jsonl` with `newThreadId -> sourceThreadId -> handoffFile`; later continuation bootstrap includes bounded lineage and tells the next Agent to read lineage handoff files before guessing about old facts, PR state, risks, or architecture decisions. Fixed private/public/README/GitHub release reminders must not be hard-coded into every continuation; they should appear only when the current workspace context or the source-thread handoff explicitly says they are relevant.
- The browser should not force-open the source thread before starting rollout continuation, because very large source threads can block on detail loading. It should start a continuation job, show job phase/status, allow the user to switch to other sessions while the job runs, restore polling from the saved job id after refresh when possible, and switch to the new continuation thread after completion. A `thread/archived` event for the source thread during continuation must not clear an unrelated current detail view.
- When the inherited runtime is full access (`danger-full-access` sandbox or root-write permission profile), Mobile Web normalizes missing/`on-request` approval policy to `never` for new turns, because persisted Desktop thread metadata can otherwise represent full filesystem access while still asking for command approval.
- Message submission is idempotent in the Mobile Web server. The browser sends `clientSubmissionId`, and `server.js` deduplicates modern clients by that id only. Content-fingerprint dedupe is retained only for legacy/no-id requests, so intentional repeated short messages are not suppressed.
- Message submission diagnostics are intentionally compact: `server.js` logs `[message-submit]` events for received/empty/done/failed submissions without raw message text, and the browser can send best-effort `/api/client-events` diagnostics for UI stalls, send stalls, no-submit button taps, and send failures. These events are local operational signals and should not contain raw secrets or long payloads.
- Mobile send UX uses explicit busy/failed button states plus a send watchdog. Slow submission should show a retry-safe status instead of looking frozen; quota/rate-limit errors should be normalized into a clear model-specific message where possible.
- Background thread completion can trigger a local browser/PWA completion tone or haptic when an unread non-current thread finishes. This complements Web Push and should be throttled so rapid status updates do not spam alerts.
- Windows Codex Desktop runs its own stdio app-server unless launched through the shared mux path.

## Product Rules

- Mobile rendering is interaction-first.
- Historical command/tool/file-change payloads are hidden.
- The latest visible turn may render at most one compact operation card.
- If app-server `thread/read` omits operation items, Mobile Web may synthesize one compact latest operation card from the thread rollout JSONL tail (`exec_command_end`, `patch_apply_end`, `function_call`, `custom_tool_call`, or Web Search events), without command output or diffs.
- Live operation cards are globally collapsed within the latest turn: only the newest command/file/tool/search operation remains visible, even if older operation cards have scrolled upward or intervening output exists. This avoids long runs of repeated tool boxes on small screens. Because only one operation card is visible, operation cards may use a compact three-line layout: the first row shows operation type with status immediately following it, and the detail area shows the command, file list, or tool/search summary across at most two clipped lines.
- File-change cards show compact file names only, not diffs or full change payloads.
- Web Search items are rendered as compact live operation cards like command/tool calls, not expanded structured payloads.
- Reasoning items are not rendered in the conversation and reasoning deltas must not remove or replace live command/file/tool operation cards.
- The left-swipe Subagent status panel is intentionally current-only: it shows `collabAgentToolCall` items from the current live turn, treating completed/closed spawn-call rows in that live turn as current because the child Agent can still be running after the spawn call closes. It must not scan older turns or show completed/failed historical Subagent records, because large collaboration sessions can otherwise produce hundreds of stale entries.
- Conversation rendering uses a visible-content signature and a lightweight keyed DOM patcher to avoid replacing the whole conversation when polling/status refreshes only change local text, item status, context-compaction status, or operation cards; this prevents no-op and broad refresh flicker.
- Thread refresh merges preserve synthetic `mux-user-*` user-message echo items when the app-server historical snapshot does not contain them, because active-turn `turn/steer` input can otherwise disappear from the visible stream after a refresh.
- If a real `userMessage` and a matching synthetic `mux-user-*` echo are both present, the real item wins and the mux echo must be dropped. This covers delayed mux notification replay after the real app-server item has already reached the browser.
- Thread refresh merges also treat matching ordinary `userMessage` items with different ids as the same visible input when they are in the same turn. This covers the case where a live local item and the later app-server snapshot have identical text or overlapping upload paths, preventing duplicate `You` cards in the browser while the durable app-server history still contains only one item. Matching user messages must be merged at their original visible position and the incoming duplicate must be marked consumed, so final turn refreshes do not append old `You` cards to the bottom.
- Turn completion and thread refresh merges must not replace locally streamed visible turn items with an empty or shorter server snapshot; preserve local visible items until an equal or fuller snapshot arrives. When a final app-server snapshot contains the same agent/plan text under a different item id, preserve the existing render identity so long outputs do not broad-reanimate at turn completion.
- Conversation card timestamps should prefer item timestamps, then explicit turn completion timestamps for completed agent/plan messages, then turn start time. Running/incomplete turns must not fall back to thread-level `updatedAt`, because continuation threads can keep stale thread summary times such as the continuation creation time.
- Uploaded images are compressed in the browser before submission where supported (`jpeg` / `png` / `webp`, default max edge 1280px and JPEG quality 0.72), saved under the upload root, and summarized as local file-path references in message text.
- Image uploads are reference-only by default for model context: `CODEX_MOBILE_IMAGE_CONTEXT_MODE=reference` sends no app-server `localImage` input parts. `latest` / `vision` sends only the latest uploaded image as a visual input, and `all` restores legacy all-image behavior. The default exists because app-server current history can retain `input_image` payloads in compacted `replacement_history` snapshots even when extended-history persistence is disabled.
- Image-upload turns are treated as temporary references for app-server extended-history persistence: Mobile Web does not request `persistExtendedHistory` for turns that include image uploads unless `CODEX_MOBILE_PERSIST_IMAGE_EXTENDED_HISTORY=1` is set. `CODEX_MOBILE_PERSIST_EXTENDED_HISTORY=0` disables the request globally. This reduces repeated `replacement_history` image payload retention, but cannot remove images already held in a currently loaded app-server in-memory thread or old rollout records.
- Uploaded non-image files are saved locally and referenced in message text by absolute path.
- Uploaded files under `%USERPROFILE%\.codex-mobile-web\uploads` can be served back to the authenticated browser through `/api/uploads/file?path=<absolute-upload-path>`; the server must only allow paths inside the upload root.
- User message rendering must show image input parts as centered thumbnails, not full-width inline previews, and must never stringify full data-URL image payloads into the conversation. Current thumbnail cap is about `min(72vw, 320px)` wide and `240px` high.
- App-server `imageView` items should render the referenced image directly in the conversation using the existing authenticated `/api/files/preview/content` path with current-thread workspace-root validation, not as raw JSON.
- The composer shows model, reasoning effort, permission, and quota as four compact runtime cards.
- Model, reasoning effort, and permission can be changed before sending. Existing-thread sends submit the selected values with the next `turn/start`; new-thread first messages submit selected values when creating and starting the first turn.
- The composer shows 5-hour and weekly quota as separate reset-aware chips after app-server emits `account/rateLimits/updated`; the chips display remaining percentage, short reset time, and normal/warn/danger coloring. Rate-limit updates are cached by model/limit id, and the UI should display the currently selected composer model's quota instead of the most recent quota event from another model.
- Composer drafts are browser-local and keyed by thread or new-thread workspace. Text lives in `localStorage`; resumable attachment drafts use IndexedDB where available and are cleared only after successful send.
- The sidebar menu header includes a compact settings button. The settings panel contains theme choices and font-size choices using the same segmented-button style, keeping both controls in the menu/sidebar area rather than the main conversation topbar.
- Font-size preference is stored in `localStorage` as `codexMobileFontSize` with values `small`, `default`, `large`, `xlarge`, or `xxlarge`. It adjusts conversation text, markdown/code/table content, approval details, and composer input through CSS variables on `:root[data-font-size]`; `xxlarge` is the large-tablet readability option above `特大`.
- Mobile Web intentionally disables browser page zoom for the app shell. `index.html` locks viewport scale to `1`, installs early non-passive WebKit `gesture*` / `dblclick` / double-touch guards, and CSS sets root `touch-action: pan-x pan-y`; readability should be handled through the in-app font-size setting.
- Theme preference is stored in `localStorage` as `codexMobileTheme` with values `system`, `dark`, or `light`; the inline head script applies `data-theme` and theme-color metadata before the main app bundle loads.
- Tablet-sized and touch layouts should not keep the sidebar permanently visible. `MENU_OVERLAY_MEDIA` / matching CSS currently uses `(max-width: 1180px), (pointer: coarse) and (max-width: 1400px)` so phones and large tablets show a full-width main pane with the sidebar opened only through the top-left menu button. The narrower `760px` breakpoint remains for phone-specific compact composer/content layout.
- On overlay layouts, a left-edge right-swipe opens the sidebar/session list. Opening the sidebar should render existing list data immediately; if the list is older than 60 seconds, refresh quietly in the background instead of blocking menu open.
- Thread rows support a long-press action sheet with rename, rollout continuation, and archive actions. Rename uses `PATCH /api/threads/<threadId>/name` and the server tries supported app-server title update methods; if none works it returns `501`.
- Agent messages and structured text blocks expose copy controls. The copy text is kept in memory for the current render, uses Clipboard API on secure contexts, and falls back to a hidden textarea/`execCommand("copy")`; do not persist copied text.
- When no thread is selected, the main pane lists recent workspaces and recent threads as shortcuts.
- Threads whose rollout JSONL reaches the configured warning threshold show a compact size warning in the thread list/home shortcuts and a current-thread banner with "跳过" and same-workspace "压缩续接" actions. If `readStateDbThread()` cannot read a per-thread summary, the detail route must fall back to app-server `thread/list` to recover the thread `cwd` before applying visible-workspace filtering.

The earlier private-only boot fallback overlay, early `early_boot` diagnostics, `/api/client-reset` recovery page, and separate `public/service-worker.js` path were intentionally dropped during the 2026-05-13 public-upstream sync. Stale active-turn cleanup is only for browser `activeTurnId` values that are superseded by newer durable turns or missing after the terminal-idle grace window, and pending server requests/items must block cleanup.

## Hermes-Codex Mux Bridge

- Codex Mobile owns the first-stage Hermes-Codex Mux polling worker for fixed worker identity `codex-hermes-main`.
- The worker service lives in `adapters/hermes-codex-mux-worker-service.js`; the CLI entrypoint is `scripts/codex-hermes-mux-worker.js`.
- Run one poll with `npm.cmd run mux:worker -- --once`; run continuous polling with `npm.cmd run mux:worker -- --base-url http://127.0.0.1:8797 --poll-ms 5000`.
- Hermes Mux API routes are owner-authenticated and accept the local header `x-hermes-web-key`. For local recovery, set `CODEX_HERMES_MUX_AUTH_HEADER_NAME=x-hermes-web-key` and `CODEX_HERMES_MUX_AUTH_VALUE_FILE=C:\ProgramData\HermesMobile\data\secrets\owner-web-key.secret`; do not print or persist the file contents.
- The worker calls `GET /api/codex-mux/tasks?assignedWorker=codex-hermes-main&status=open,running`, `GET /api/codex-mux/tasks/:taskId`, `GET /api/codex-mux/tasks/:taskId/events`, `POST /api/codex-mux/tasks/:taskId/events`, and `POST /api/codex-mux/workers/codex-hermes-main/heartbeat`.
- Current restored background worker logs are under `%USERPROFILE%\.codex-mobile-web\hermes-codex-mux-worker-restored.out.log` and `.err.log`.
- Manual Codex Mobile shared-chain restart does not guarantee this separate Hermes-Codex polling worker is running. If Hermes Mobile stops receiving Codex Mux updates after a restart, first check the worker process, log tail, and Hermes heartbeat for `codex-hermes-main`.

## Safety

- Do not patch the WindowsApps Codex installation.
- Do not replace Codex Desktop's startup command unless explicitly requested and tested through a reversible path.
- Prefer the reversible `start-codex-desktop-shared.ps1` launcher over persistent system environment changes for Desktop/Web live sync.
- Keep `.codex` state read-only except through app-server RPCs.
- Keep raw access keys out of Git and out of `.agent-context`.
