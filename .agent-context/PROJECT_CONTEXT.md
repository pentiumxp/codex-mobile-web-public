## Product Rules

- Mobile rendering is interaction-first.
- Historical command/tool/file-change payloads are hidden.
- The latest visible live turn may render at most one compact operation card. Completed turns should not keep command/tool/file/search operation cards below the final reply; when scoped Usage data exists, the final frame should be `turnUsageSummary`.
- If app-server `thread/read` omits operation items, Mobile Web may synthesize one compact latest live operation card from the thread rollout JSONL tail (`exec_command_end`, `patch_apply_end`, `function_call`, `custom_tool_call`, or Web Search events), without command output or diffs. Completed raw fallback is allowed only while the latest turn is still live and the operation is tied to that same turn id, so older completed operations cannot attach to newer live turns.
- Latest live-turn operation cards are globally collapsed within the latest turn: only the newest command/file/tool/search operation remains visible, even if older operation cards have scrolled upward or intervening output exists. This avoids long runs of repeated tool boxes on small screens. Because only one operation card is visible, operation cards may use a compact four-line layout: the first row shows operation type with status immediately following it, and the detail area shows the command, file list, or tool/search summary across at most three clipped lines.
- File-change cards show compact file names only, not diffs or full change payloads.
- Web Search items are rendered as compact live operation cards like command/tool calls, not expanded structured payloads.
- Reasoning items are not rendered in the conversation and reasoning deltas must not remove or replace live command/file/tool operation cards.
- The left-swipe Subagent status panel is intentionally current-only: it shows `collabAgentToolCall` items from the current live turn, treating completed/closed spawn-call rows in that live turn as current because the child Agent can still be running after the spawn call closes. It must not scan older turns or show completed/failed historical Subagent records, because large collaboration sessions can otherwise produce hundreds of stale entries.
- Conversation rendering uses a visible-content signature and a lightweight keyed DOM patcher to avoid replacing the whole conversation when polling/status refreshes only change local text, item status, context-compaction status, or operation cards; this prevents no-op and broad refresh flicker.
- Thread refresh merges preserve synthetic `mux-user-*` user-message echo items when the app-server historical snapshot does not contain them, because active-turn `turn/steer` input can otherwise disappear from the visible stream after a refresh.
- If a real `userMessage` and a matching synthetic `mux-user-*` echo are both present, the real item wins and the mux echo must be dropped. This covers delayed mux notification replay after the real app-server item has already reached the browser.
- Thread refresh merges also treat matching ordinary `userMessage` items with different ids as the same visible input when they are in the same turn. This covers the case where a live local item and the later app-server snapshot have identical text or overlapping upload paths, preventing duplicate `You` cards in the browser while the durable app-server history still contains only one item. Matching user messages must be merged at their original visible position and the incoming duplicate must be marked consumed, so final turn refreshes do not append old `You` cards to the bottom.
- Turn completion and thread refresh merges must not replace locally streamed visible turn items with an empty or shorter server snapshot; preserve local visible items until an equal or fuller snapshot arrives. When a final app-server snapshot contains the same agent/plan text under a different item id, preserve the existing render identity so long outputs do not broad-reanimate at turn completion.
- Conversation card timestamps should prefer item timestamps, then explicit turn completion timestamps for completed agent/plan messages, then turn start time. Running/incomplete turns must not fall back to thread-level `updatedAt`, because continuation threads can keep stale thread summary times such as the continuation creation time.
- Uploaded images are compressed in the browser before submission where supported (`jpeg` / `png` / `webp`, default max edge 1280px and JPEG quality 0.72), saved under the upload root, and summarized as local file-path references in message text.
- Image uploads are reference-only by default for model context: `CODEX_MOBILE_IMAGE_CONTEXT_MODE=reference` sends no app-server `localImage` input parts. `latest` / `vision` sends only the latest uploaded image as a visual input, and `all` restores legacy all-image behavior. The default exists because app-server current history can retain `input_image` payloads in compacted `replacement_history` snapshots even when extended-history persistence is disabled.
- Image-upload turns are treated as temporary references for app-server extended-history persistence: Mobile Web does not request `persistExtendedHistory` for turns that include image uploads unless `CODEX_MOBILE_PERSIST_IMAGE_EXTENDED_HISTORY=1` is set. `CODEX_MOBILE_PERSIST_EXTENDED_HISTORY=0` disables the request globally. This reduces repeated `replacement_history` image payload retention, but cannot remove images already held in a currently loaded app-server in-memory thread or old rollout records.
